# 1. Purpose

Define a pluggable entitlement and billing system that:

- Determines what a tenant is allowed to do
- Enables multiple entitlement sources (Azure Marketplace, licence file, etc.)
- Supports fully offline standalone operation
- Enforces limits locally in the Agent
- Maintains ControlPlane as the authoritative source of truth
- Preserves existing architectural constraints:
  - Agent executes
  - ControlPlane coordinates
  - Package remains authoritative

---

# 2. Governing Constraints

1. Entitlement decisions MUST originate from ControlPlane.
2. Agents MUST NOT call billing or entitlement providers directly.
3. Agents MUST enforce limits locally using a snapshot.
4. ControlPlane MUST remain authoritative for:
   - job admission
   - usage tracking
   - revocation
5. Entitlement evaluation MUST be deterministic.
6. Billing usage recording MUST be idempotent.
7. No module may reference entitlement or billing logic.
8. All enforcement MUST occur at:
   - job submission (ControlPlane)
   - lease boundaries (ControlPlane)
   - unit-of-work boundaries (Agent)
9. The system MUST operate without network connectivity in Standalone mode.
10. All entitlement inputs MUST be cryptographically verifiable.
11. Usage MUST NOT be trusted from the Agent; ControlPlane reconstructs authoritative totals from checkpoint cursors.
12. There MUST be no implicit or environment-based entitlement behaviour (e.g. "unlimited").

---

# 3. Concepts

## 3.1 Entitlement

Defines allowed behaviour:

- Feature flags (enable/disable capabilities)
- Limits (absolute caps per module: WorkItems, Pipelines, Repos, etc.)
- Enforcement behaviour (Block, Warn, Audit)

> Note: “remaining quota” is derived at runtime from usage, not stored in the entitlement source.

---

## 3.2 Billing

Tracks and charges for usage. May be:

- Absent (community / OSS — capped but full-featured)
- Licence-based (offline or connected)
- Marketplace metered (Azure Marketplace SaaS fulfilment API)

Billing is an **external concern**. Entitlement is the enforcement mechanism.

---

## 3.3 Entitlement Snapshot

Immutable contract delivered to Agent inside the lease response:

- Generated by ControlPlane
- Used locally for enforcement
- Time-bound (TTL)
- Refreshed with each heartbeat response
- Derived from:
  - entitlement source (licence / marketplace)
  - local usage ledger (checkpoint cursor counts)

---

## 3.4 Entitlement Source

Origin of entitlement data:

- Azure Marketplace SaaS (online)
- Licence server (online)
- Signed licence file (offline)
- Community (local, capped)

All sources MUST produce equivalent logical output.

---

## 3.5 Entitlement Scope

Entitlement may be scoped at multiple levels:

| Level | Use case |
|---|---|
| **Tenant** | Organisation-wide entitlement (primary) |
| **Tenant + User** | Personal limits within a tenant pool |
| **Contract/Subscription** | Multiple independent contracts per tenant (e.g. consulting firm with per-customer entitlements) |

A consulting company operating on behalf of customers MUST be supported via both models:
- Consultant's own licence covering all client work
- Per-customer entitlement operated by the consultant

# 4. Interfaces

## 4.1 EntitlementContext

```csharp
public sealed record EntitlementContext
{
    /// <summary>Provider-specific identifier declared in config:
    /// licence key, marketplace subscription ID, or tenant GUID.</summary>
    public required string EntitlementId;

    /// <summary>Operator's tenant (from auth token).</summary>
    public required string TenantId;

    /// <summary>Optional: specific contract when operating on behalf of a customer.</summary>
    public string? ContractId { get; init; }
}
```

The entitlement identity is declared in the configuration file, not derived from the authentication token. This decouples operator identity from billing identity.

---

## 4.2 IEntitlementProvider

```csharp
public interface IEntitlementProvider
{
    Task<EntitlementSourceData> GetEntitlementAsync(
        EntitlementContext context,
        CancellationToken ct);

```

> Providers return raw entitlement data, not a snapshot. Snapshot construction is the responsibility of ControlPlane.

## 4.4 UsageDelta

```csharp

{

    public required string Module;

```

Usage recording is idempotent: the combination of `JobId + Module + Cursor` is the natural key. Safe to replay.

# 5. Entitlement Snapshot Model

```csharp
public sealed record EntitlementSnapshot
{
    public required string Source;
    public required string Plan;

    public required EntitlementMode Mode;

    public required IReadOnlyDictionary<string, FeatureEntitlement> Features;
    public required IReadOnlyDictionary<string, QuantityEntitlement> Quantities;

    public required DateTimeOffset IssuedAt;
    public required DateTimeOffset ExpiresAt;

    public required string Signature;
}

public enum EntitlementMode
{
    Enforce,
    WarnOnly,
    AuditOnly,
    Deny
}

public sealed record FeatureEntitlement
{
    public required bool Enabled;
    public required EnforcementBehaviour Behaviour;
}

public sealed record QuantityEntitlement
{
    public required long Limit;
    public required long Used;
    public required EnforcementBehaviour Behaviour;
}

public enum EnforcementBehaviour
{
    Block,
    Warn,
    Audit
}
```

# 6. Snapshot Delivery & Refresh

## 6.1 Initial Delivery

The `EntitlementSnapshot` is included in the lease response from `GET /agents/lease`. The agent receives it alongside the `MigrationJob` — no separate call required.

## 6.2 TTL Refresh

The snapshot has a bounded TTL. On each heartbeat (`POST /agents/lease/{leaseId}/heartbeat`), the control plane returns an updated snapshot in the response body. This ensures:

- Long-running jobs always have a current snapshot
- Revocation propagates within one heartbeat interval
- No separate refresh endpoint is needed

## 6.3 Revocation

If entitlement is exhausted or revoked between heartbeats, the control plane returns a snapshot with `Mode = Deny` in the heartbeat response. The agent MUST check the snapshot mode after each heartbeat and stop gracefully at the next checkpoint boundary.

# 7. Offline Licensing (Standalone Mode)

## 7.1 Licence File

Standalone mode MUST support a signed licence file:

* JSON payload
* RSA signature
* Optional encryption (not required)

Example:

```json
{
  "entitlementId": "lic-customer-123",
  "tenantId": "customer-123",
  "plan": "Enterprise",
  "issuedAt": "2026-01-01T00:00:00Z",
  "expiresAt": "2027-01-01T00:00:00Z",
  "limits": {
    "WorkItems.Processed": 500000,
    "Pipelines.Processed": 1000,
    "Repos.Processed": 500
  },
  "features": {
    "HostedAgents": false
  },
  "signature": "base64..."
}
```

## 7.2 Validation Rules

ControlPlane MUST:

1. Verify signature using embedded public key
2. Validate time window
3. Reject invalid or missing licence

Failure behaviour:

| Condition         | Behaviour    |
| ----------------- | ------------ |
| Missing licence   | Deny         |
| Invalid signature | Deny         |
| Expired           | Deny or Warn |
| Corrupt           | Deny         |

---

## 7.3 Cryptographic Requirement

* MUST use asymmetric signature (RSA or equivalent)
* MUST NOT rely on encryption alone
* Public key MUST be embedded in product binaries

# 8. Usage Model

## 8.1 Source of Truth

Usage is reconstructed from **checkpoint cursors** — the authoritative record of what has been processed.

- Hosted mode: ControlPlane PostgreSQL database
- Standalone mode: Local SQLite database (same schema, different backing store)

Agent-reported usage is treated as advisory only.

---

## 8.2 Usage Calculation

```csharp
remaining = limit - totalUsed;
```

`totalUsed` is reconstructed from checkpoint cursor positions per module.

---

## 8.3 Idempotency

Usage recording MUST be idempotent:

* Natural key: `jobId + module + cursor`
* Safe to replay

---

## 8.4 Quantity Meters

Entitlements can cap per-module processed counts:

| Meter | Description |
|---|---|
| `WorkItems.Processed` | Total work items processed across all jobs |
| `Pipelines.Processed` | Total pipelines processed |
| `Repos.Processed` | Total repositories processed |
| `Teams.Processed` | Total teams processed |
| `Agents.Concurrent` | Maximum concurrent agents |

Each module defines its own meter. New modules register new meters.

---

# 9. Enforcement Model

## 9.1 ControlPlane Enforcement

### Job Admission

```csharp
if (!entitlement.CanStart(job))
    reject;
```

### Lease Renewal

```csharp
if (limitExceeded)
    return snapshot with Mode = Deny;
```

---

## 9.2 Agent Enforcement

At unit-of-work boundaries (defined by each module's checkpoint cursor granularity):

```csharp
if (!snapshot.CanProcess(nextBatch))
    stop gracefully at checkpoint;
```

The agent finishes the current batch/window, writes the checkpoint, then pauses the job. It does NOT hard-stop mid-batch.

---

## 9.3 Enforcement Behaviour

| Behaviour | Meaning |
|---|---|
| Block | Graceful stop at next checkpoint |
| Warn | Continue with structured warning log |
| Audit | Record only (no user-visible effect) |

# 10. Provider Implementations

```text
IEntitlementProvider
├── AzureMarketplaceProvider   (Azure Marketplace SaaS fulfilment API)
├── LicenceServerProvider      (online licence server)
├── LicenceFileProvider        (signed offline licence file)
├── CommunityProvider          (capped, full-featured, no purchase required)
```

Resolution order is deterministic and configurable:

```text
Composite → priority-ordered list → first valid provider wins
```

---

# 10. Standalone Behaviour

Standalone mode MUST:

* operate without network
* use LicenceFileProvider
* maintain local usage ledger
* enforce all limits identically to hosted mode

There is no special-case “offline logic” beyond provider selection.

---

# 11. Prohibited Patterns

The following are explicitly disallowed:

* Environment-based entitlement branching
* Unlimited entitlement defaults
* Agent-side licence validation
* Trusting client-reported usage
* Encryption without signature

---

# 12. Summary

The system defines:

* Entitlement as a deterministic, signed input
* ControlPlane as the local or remote authority
* Agent as an execution enforcer
* Usage as locally reconstructed state

Offline and online modes differ only in **entitlement source**, not behaviour.
