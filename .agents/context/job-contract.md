# Job Contract

## Migration Job Definition

A **Migration Job** (`MigrationJob`) is the internal, serialisable unit of execution. It is the artefact that the control plane creates from the operator's config and passes to a Migration Agent. The config file is never passed directly to the agent.

The CLI constructs the `MigrationJob` after its pre-validation pass (Tier 0 structural + Tier 1 connectivity) clears. The CLI assigns the `jobId`, normalises the package URI, and computes the `configHash` before submitting to the control plane. The control plane performs a final deduplication and schema check then assigns the job to an agent. The agent receives only the `MigrationJob` â€” it has no access to the original config file.

See [docs/validation.md](validation.md) for the full four-tier validation model.

## Fields

| Field | Required | Description |
|---|---|---|
| `jobId` | Yes | Globally unique identifier for the job (UUID v4). Generated by the CLI after pre-validation passes. |
| `configVersion` | Yes | `"2.0"` for all new jobs. Used by the Job Engine for compatibility checks. |
| `mode` | Yes | `Export`, `Prepare`, `Import`, or `Migrate`. |
| `sourceType` | Yes | Type discriminator for capability-based agent routing: `AzureDevOpsServices`, `TeamFoundationServer`, or `Simulated`. Set by the CLI from `config.Source?.Type` at job construction time. |
| `package.packageUri` | Yes | URI of the package root. Must be a URI â€” `file:///` for local, or a standard Azure Blob Storage HTTPS URL (`https://<account>.blob.core.windows.net/<container>/<prefix>`) for cloud. Local filesystem paths are normalised to `file:///` by the CLI before job construction. |
| `package.createPackage` | No | If `true`, pack after export or unpack before import. Default `false`. |
| `guardrails.streamingRequired` | Yes | Must be `true`. The Job Engine will reject any execution plan that would load all revisions into memory. |
| `guardrails.canonicalWorkItemsLayoutRequired` | Yes | Must be `true`. Any module that would alter the canonical folder structure is rejected at plan time. |
| `diagnostics.minimumLevel` | No | Minimum log level for the agent's diagnostic output. |
| `resume.mode` | No | `Auto` (default) â€” detect existing cursor and resume from last position; `ForceFresh` â€” delete all module cursor files and `.migration/Checkpoints/job.phase.json` before running. Absent or `null` is treated as `Auto`. |

## Source/Target Endpoint Shapes

The `source` and `target` fields are **polymorphic** â€” the `type` discriminator determines which concrete shape is used. The `PolymorphicEndpointOptionsConverter` reads the `type` field first and deserialises to the matching concrete class.

### AzureDevOpsServices

```json
{
  "type": "AzureDevOpsServices",
  "url": "https://dev.azure.com/myorg",
  "project": "MyProject",
  "apiVersion": "7.1",
  "authentication": {
    "accessToken": "..."
  }
}
```

| Field | Required | Description |
|---|---|---|
| `type` | Yes | Must be `AzureDevOpsServices`. |
| `url` | Yes | Organisation URL. Supports `${...}` token expansion. |
| `project` | Yes | Team project name. |
| `apiVersion` | No | REST API version (default: latest). |
| `authentication.accessToken` | Yes | PAT or service principal token. |

### TeamFoundationServer

```json
{
  "type": "TeamFoundationServer",
  "url": "http://tfsserver:8080/tfs/DefaultCollection",
  "project": "MyProject",
  "authentication": {
    "accessToken": "..."
  }
}
```

### Simulated

```json
{
  "type": "Simulated",
  "generator": {
    "workItemCount": 50,
    "revisionCount": 3,
    "maxAttachments": 2,
    "maxLinks": 1
  }
}
```

| Field | Required | Description |
|---|---|---|
| `type` | Yes | Must be `Simulated`. |
| `generator` | No | Controls the shape and volume of generated test data. |

## URI Normalisation

The `artefacts.packageUri` field must always be a URI, not a bare path. The TUI normalises the value from the config file's `artefacts.workingDirectory` field:

| Config value | Normalised `packageUri` |
|---|---|
| `D:\exports\run-001` | `file:///D:/exports/run-001` |
| `/home/user/exports/run-001` | `file:///home/user/exports/run-001` |
| `file:///D:/exports/run-001` | unchanged |
| `https://<account>.blob.core.windows.net/<container>/<prefix>` | unchanged |

Only `file:///` is required in Phase 1. The URI shape must accommodate standard Azure Blob Storage HTTPS URLs without schema changes.

## Relationship to Configuration File

The job contract and the local configuration file ([docs/configuration.md](configuration.md)) share field semantics. The **CLI** converts between them (the TUI is a pure progress viewer and never builds job contracts):

| Aspect | Local config file | MigrationJob |
|---|---|---|
| Package location | `artefacts.workingDirectory` (bare path) | `artefacts.packageUri` (URI) |
| Identity | None | `jobId` (tracked by control plane in remote mode) |
| Guardrails | Implicit | Explicit fields in `guardrails` |
| Secrets | Referenced directly or via env vars | Passed through the job definition as configured |

## Guardrails

The `guardrails` block encodes non-negotiable constraints that the Job Engine enforces before execution begins:

- `streamingRequired: true` â€” the Job Engine will fail-fast if the execution plan requires loading more than one revision folder into memory simultaneously.
- `canonicalWorkItemsLayoutRequired: true` â€” the Job Engine will reject any module configuration that alters the `WorkItems/yyyy-MM-dd/<ticks>-<workItemId>-<revisionIndex>/` layout.

These fields exist in the job contract so that guardrail enforcement is explicit and auditable, not implicit. They must always be `true`; a job with either field set to `false` is rejected by the Job Engine.

## Secrets Handling

Credentials are configured by the operator in the scenario config file and passed through the `MigrationJob` definition. The control plane persists `job_json` (which contains the credentials) but does not inspect or proxy credential values.

```json
"source": {
  "type": "AzureDevOpsServices",
  "url": "https://dev.azure.com/myorg",
  "project": "MyProject",
  "apiVersion": "7.1",
  "authentication": {
    "credential": {
      "type": "PersonalAccessToken",
      "pat": "<token>"
    }
  }
}
```

Migration Agents read credentials directly from the job definition at startup. In local mode, credentials may also be supplied via environment variables.

## Two-Job Pattern (Export â†’ Import Split)

When an explicit approval gate is required between export and import, or when the two phases run in different network zones, the recommended pattern is three jobs sharing the same `packageUri`:

```
Job 1: mode=Export, artefacts.packageUri=https://<account>.blob.core.windows.net/...
        â†’ Control plane marks job 1 Complete
        â†’ Operator reviews package
Job 2: mode=Prepare, artefacts.packageUri=https://<account>.blob.core.windows.net/... (same URI)
        â†’ Agent reads exported package, queries target, writes validation artefacts
        â†’ Control plane marks job 2 Complete
        â†’ Operator reviews prepare-report.json files, edits mapping.json if needed
Job 3: mode=Import, artefacts.packageUri=https://<account>.blob.core.windows.net/... (same URI)
        â†’ Migration Agent reads package written by Job 1, validated by Job 2
```

The package URI is the handoff point. The `manifest.json`, cursor state, `idmap.db`, and Prepare artefacts written by Jobs 1 and 2 are read by Job 3 without modification. If Prepare has not been run (no `.migration/Checkpoints/prepare.complete.json`), Import auto-runs Prepare and aborts on blocking issues.

See [docs/migration-agent.md](migration-agent.md) for Migration Agent behaviour in each pattern.

### Mode Semantics Summary

| Mode | Requires Source | Requires Target | Requires Package | Description |
|------|----------------|-----------------|------------------|-------------|
| `Export` | Yes | No | Writes to | Extract data from source into the package. |
| `Prepare` | No | Yes | Reads from, writes validation artefacts | Cross-validate exported package against target. Produces mapping and validation reports for operator review. |
| `Import` | No | Yes | Reads from | Import data from package into target. Auto-runs Prepare if not done; aborts on blocking issues. |
| `Migrate` | Yes | Yes | Writes to, then reads from | Export â†’ Prepare â†’ Import in one run. Aborts after Prepare on blocking issues. |
