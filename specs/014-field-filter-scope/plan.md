# Implementation Plan: Field Filter Scope for Work Items

**Branch**: `014-field-filter-scope` | **Date**: 2026-04-17 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/014-field-filter-scope/spec.md`

## Summary

Add a `filter` scope type to the `WorkItems` module (export, import, inventory, and discovery) that includes or excludes work items based on a regex match against a named ADO field. Add a `Scopes` array to `OrganisationEntry` and `DiscoveryJobOrganisation` to support `wiql` and `filter` scopes for all scoped operations. All new types live in `DevOpsMigrationPlatform.Abstractions`; all evaluation logic lives in `DevOpsMigrationPlatform.Infrastructure`. No new projects required.

## Technical Context

**Language/Version**: C# 12 / .NET 10  
**Primary Dependencies**: MSTest, Reqnroll, Moq — all already present  
**Storage**: `IArtefactStore` (existing) — no new storage  
**Testing**: MSTest + Reqnroll; unit tests in `WorkItemFieldFilterTests.cs`, `WorkItemsModuleExtensionsTests.cs`; BDD steps for acceptance scenarios

**Performance Goals**: Filtering adds zero API calls per item; export filter evaluated on current work item state before revision history is fetched (filtered-out items incur no revision API calls); import filter uses a lightweight folder-name pre-pass (no file reads in pre-pass); inventory and discovery filter evaluation uses the full work item already loaded; regex evaluated per item with 2-second internal timeout  
**Constraints**: `IWorkItemDiscoveryService` receives WIQL from org scope; `WorkItemExportOrchestrator` receives filter chain; `WorkItemImportOrchestrator` receives filter chain  
**Scale/Scope**: Works at up to 25,000 work items (existing scale target)

## Design Decisions

### WorkItemFieldFilter evaluation point — export

The filter chain is passed to `WorkItemExportOrchestrator` as an optional constructor parameter `IReadOnlyList<WorkItemFieldFilterOptions>? filters`. Filter evaluation occurs **before requesting revision history** — the current work item state (fields already present in the source stream) is evaluated against the filter chain. If the work item fails the filter, no revision history API call is made, eliminating wasted API calls for filtered-out items.

All ADO fields from the current work item object are available. A `HashSet<int>` of filtered work item IDs tracks skipped items in case the same ID is encountered again in the stream.

### WorkItemFieldFilter evaluation point — import

The filter is evaluated against the **last revision** in the package (current state of the work item). A lightweight pre-pass enumerates revision folder names only — no file reads — to identify the last folder per work item ID. Folder names encode the work item ID and revision index, so this is a pure directory listing with no I/O beyond stat calls.

After the pre-pass, only the `revision.json` files from the identified last-revision folders are loaded — one file read per distinct work item. These field values are evaluated against the filter chain, building a `HashSet<int>` of filtered work item IDs before the main streaming import pass begins.

The main streaming pass skips all revision folders for IDs in the filtered set. This approach is streaming-safe (the set is bounded by distinct work items, not revisions) and evaluates against the work item's current/final state.

### IWorkItemDiscoveryService interface change (breaking)

`IWorkItemDiscoveryService.DiscoverWorkItemsAsync` gains a new optional parameter:

```csharp
IAsyncEnumerable<ProjectDiscoverySummary> DiscoverWorkItemsAsync(
    string url,
    string project,
    string pat,
    IReadOnlyList<WorkItemFieldFilterOptions>? filterOptions = null,
    CancellationToken cancellationToken = default);
```

This is a controlled breaking change — the platform owns all implementations. Implementations:

- **`AzureDevOpsWorkItemDiscoveryService`**: when `filterOptions` is non-empty, unions field names into the `fields:` array of `GetWorkItemsAsync`, evaluates `WorkItemFieldFilterChain` per item, skips non-matching items from the summary count.
- **`TfsObjectModelWorkItemDiscoveryService`**: accepts the parameter but ignores it — TFS inventory filter is out of scope; the parameter is present only to satisfy the interface contract.

`InventoryService.RunInventoryAsync` builds a `WorkItemFieldFilterChain` from the org's `filter` scopes and passes it as `filterOptions` to `DiscoverWorkItemsAsync`.

### WorkItemFieldFilter evaluation point — inventory and discovery

`filter` scope evaluation in inventory and discovery uses the existing `GetWorkItemsAsync` batch call that already fetches `System.Rev`. When filter scopes are configured on an organisation, the filter-referenced field reference names are **unioned into the `fields:` array** of that existing call:

```csharp
// Before: fields: new[] { "System.Rev" }
// After (when filters present):
var fields = new HashSet<string> { "System.Rev" };
foreach (var f in filterOptions) fields.Add(f.Field);
var workItems = await witClient.GetWorkItemsAsync(batch, fields: fields.ToArray(), ...);
```

This adds **zero extra API calls** — only a marginally larger response payload per batch. Field values are then available for `WorkItemFieldFilterChain.Passes()` in-process.

> **Performance note**: Each filter scope field name added to the `fields:` array increases the batch response payload. Batch size is fixed at 200 items. Large text fields (e.g. `System.Description`, `System.History`) can multiply payload size significantly at scale. Users should be advised (in `docs/configuration.md`) to: (1) use the minimum number of filter scopes needed; (2) prefer short reference-data fields (`System.AreaPath`, `System.WorkItemType`, `System.State`) over large text fields.

- **Discovery/Dependency** (`DependencyDiscoveryService` → `IWorkItemLinkAnalysisService.AnalyseLinksAsync`): uses a separate service path. The same pattern applies — add filter field names to whatever batch field fetch exists in that path. Verified during T020a.
- **WIQL SELECT clause**: The WIQL query returns only work item IDs (by design — `SELECT [System.Id]`). Adding fields to the WIQL SELECT does not return field values; the `GetWorkItemsAsync` batch is the correct injection point.

### Zero-match warning

After export/import completes, if `workItemsProcessed == 0` and at least one filter was configured, a warning is emitted via `IProgressSink` with `Message = "[WorkItems] Warning: all work items were filtered out by filter scopes. Check your filter configuration."`.

### Organisation wiql scope

`OrganisationEntry` gains `List<MigrationOptionsScope> Scopes`. `InventoryCommand` maps these to `List<JobModuleScope> Scopes` on `DiscoveryJobOrganisation`. `InventoryServiceFactory.BuildDiscoveryOptions()` reads the `wiql` scope from each `DiscoveryJobOrganisation` and sets `BaseQuery` on the `WorkItemQueryWindowOptions`. `WorkItemQueryWindowStrategy.EnumerateWindowsAsync` already accepts `options.BaseQuery` — this is a plumbing-only change.

### Validation

A new static `WorkItemFieldFilterOptions.Validate()` method throws `InvalidOperationException` with a descriptive message if:
- `Mode` is not `"include"` or `"exclude"` (case-insensitive)
- `Field` is null or empty
- `Pattern` is not a valid .NET regex

Validation is called from `WorkItemsModuleExtensions.FromModule()` immediately after parsing each filter scope. This causes the CLI to fail fast before the job is submitted.

### Diagnostic log levels for filter skips

When a work item is skipped by a filter scope, the log entry uses structured log levels:

- **Information** (always): `"Work item {id} skipped — filter field='{field}' mode='{mode}'"`
- **Debug** (when `ILogger.IsEnabled(LogLevel.Debug)`): appends the actual field value
- **Trace** (when `ILogger.IsEnabled(LogLevel.Trace)`): appends the pattern

Using `IsEnabled()` guards avoids string allocation for interpolated values when the log level is not active. Both `WorkItemExportOrchestrator` and `WorkItemImportOrchestrator` (and inventory/discovery call sites) use this pattern.

### Regex timeout

`WorkItemFieldFilter.Passes()` uses `Regex.IsMatch(value, pattern, options, TimeSpan.FromSeconds(2))`. `RegexMatchTimeoutException` is **not caught in `WorkItemFieldFilter`** — it propagates to `WorkItemFieldFilterChain.Passes()`, which catches it, logs a warning (`"Regex timeout evaluating field='{field}' — treating as non-match"`), and returns `false` for `include` mode or `true` for `exclude` mode. Migration continues uninterrupted. This keeps `IWorkItemFieldFilter` dependency-free (no `ILogger`).
